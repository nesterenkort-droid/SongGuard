# TrackGuard — руководство по эксплуатации

Этот документ — на случай, если что-то пошло не так, а рядом нет программиста.
Все команды выполняются на сервере (VPS), в папке проекта (`cd /opt/trackguard` или
где он у вас развёрнут), если не сказано иное.

## Быстрая проверка «всё ли работает»

Открой в браузере `https://<твой-домен>/` — если видна панель, значит веб-сервер
жив. Подробный статус компонентов: `https://<твой-домен>/healthz` (JSON) или
страница дашборда (плитки здоровья вверху).

Через терминал:
```bash
docker compose ps
```
Все сервисы должны быть `Up` (для `postgres`/`redis`/`web` — ещё и `(healthy)`).
`migrate` и `seed` — одноразовые, у них будет статус `Exited (0)`, это нормально.

## Перезапуск

Перезапустить всё:
```bash
docker compose restart
```
Перезапустить один сервис (например, если завис бот):
```bash
docker compose restart bot
```
Если после `git pull` что-то не поднялось — пересоберите образ:
```bash
docker compose up -d --build
```

## Логи

Логи одного сервиса за последний час:
```bash
docker compose logs --since 1h web
docker compose logs --since 1h worker
docker compose logs --since 1h bot
```
Смотреть логи в реальном времени (Ctrl+C — выйти):
```bash
docker compose logs -f worker
```
Если в PowerShell/консоли вывод докера кажется «красным» и похожим на ошибку —
это может быть просто вывод в stderr (например, «Creating», «Started»), не сама
ошибка. Смотрите текст сообщения, а не цвет.

## Что означают алерты в Telegram

Бот присылает админам сообщение **только когда статус системы меняется**
(а не на каждой проверке) — так что если сообщение пришло, что-то реально
изменилось.

| Сообщение | Что случилось | Что делать |
|---|---|---|
| 🔴 Критическая проблема: База данных | PostgreSQL не отвечает | Проверить `docker compose ps postgres`, `docker compose logs postgres`. Если контейнер упал — `docker compose up -d postgres` |
| 🔴 Критическая проблема: Redis | Redis не отвечает | Аналогично — `docker compose restart redis` |
| 🟡 Есть проблема: Воркер (сканер) | Воркер не отчитывался дольше нормы — сканы не идут | `docker compose logs worker`, `docker compose restart worker` |
| 🟡 Есть проблема: Telegram-бот | Бот не отчитывался — карточки и команды не работают | Проверьте `TELEGRAM_BOT_TOKEN` в `.env`, `docker compose logs bot` |
| ✅ Всё снова в порядке | Проблема выше устранилась сама или после ваших действий | Ничего делать не нужно |
| ⏰ Прошло 14 дней с отправки жалобы | Площадка не ответила на жалобу | Зайдите на сайт → Находки → Пакеты, проверьте статус вручную, при необходимости подайте жалобу повторно |
| 🎉 Победа! Пиратка удалена | Система перепроверила и убедилась, что пиратский релиз пропал | Ничего делать не нужно — это хорошая новость |

Если алертов нет вообще, а сайт не открывается — см. следующий раздел
(«внешний сторож»): значит упал весь сервер целиком, и бот об этом сказать не может.

## Внешний сторож (dead-man switch)

Бот живёт на том же сервере, что и всё остальное — если сервер упадёт целиком
(не только приложение, а вся машина), бот тоже замолчит и предупредить не сможет.
Поэтому настроен внешний бесплатный сервис **healthchecks.io**: сервер обязан
пинговать его каждые несколько минут; если пинг пропадает — healthchecks.io сам
пришлёт вам email/уведомление. Если вы получили письмо от healthchecks.io
«ping missed» — это значит сервер недоступен, нужно зайти на хостинг-панель VPS
и проверить его физически (перезагрузка через панель провайдера).

## Резервные копии и восстановление

Бэкапится каждую ночь: база данных (`pg_dump`) и файлы (оригиналы треков,
evidence-архивы для жалоб) через `restic`.

**Проверить, что бэкапы вообще есть и свежие:**
```bash
bash scripts/backup.sh --list
```
Если последний бэкап старше 26 часов — придёт алерт; разберитесь, почему
`backup.sh` не отработал в cron (см. `docker compose logs` или системный cron-лог).

**Восстановление из бэкапа** (например, после серьёзной поломки БД):
```bash
bash scripts/restore.sh <ID_снапшота_или_latest>
```
Скрипт спросит подтверждение перед тем, как перезаписать текущие данные — это
разрушительное действие, отменить нельзя. Полная инструкция и пример реального
прогона восстановления — в [`docs/backup-restore.md`](docs/backup-restore.md).

⚠️ **Непроверенный бэкап — это не бэкап.** Раз в несколько месяцев стоит
по-настоящему прогнать восстановление (например, на тестовом сервере) и
убедиться, что данные действительно возвращаются.

## Вернуть себе админку

Если админский Telegram-аккаунт потерян (сменили телефон, удалили аккаунт и
т.п.), можно вручную назначить нового админа по Telegram ID:
```bash
docker compose run --rm web python -m scripts.promote <ваш_telegram_id> "Ваше имя"
```
Свой Telegram ID можно узнать у бота `@userinfobot` в Telegram.

## Диск заполняется

На плитках здоровья появится алерт при заполнении диска свыше 80%. Что чистить
в первую очередь:
- `docker compose exec worker du -sh /data/*` — посмотреть, что занимает место
  (обычно это `/data/audio` — аудио кандидатов на проверке).
- Аудио кандидатов удаляется автоматически после вердикта (retention job), но
  если что-то накопилось — можно почистить вручную папку `/data/audio` для
  треков со статусом `dismissed`/`confirmed` старше недели.
- Логи докера тоже могут расти: `docker system df` покажет, сколько места
  занимают образы/контейнеры; `docker system prune` освобождает неиспользуемое
  (безопасно, не трогает работающие контейнеры и volumes).

## Обновление до новой версии кода

```bash
bash update.sh
```
Скрипт делает `git pull`, пересобирает образы и накатывает миграции. Если после
обновления что-то сломалось:
```bash
bash rollback.sh <предыдущий_git_commit_или_тег>
```

## Панако (аудио-сравнение) не отвечает

Если в дашборде долго не появляются аудио-совпадения — проверьте:
```bash
docker compose logs worker | grep -i panako
```
Панако работает внутри контейнера `worker` (Java-процесс), не отдельным
сервисом. Перезапуск воркера (`docker compose restart worker`) обычно чинит
зависшие процессы.

## Контакты / куда смотреть в непонятной ситуации

1. Плитки здоровья на главной странице сайта.
2. `docker compose logs <сервис>` — что случилось перед сбоем.
3. Этот документ.
4. Если ничего не помогло — обратитесь за помощью к разработчику с
   логами (`docker compose logs --since 6h > problem.log`) и описанием, что
   произошло.
