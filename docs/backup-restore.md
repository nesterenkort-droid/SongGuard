# Бэкапы и восстановление

## Как это устроено

Каждую ночь (по cron на VPS) `scripts/backup.sh`:
1. Снимает `pg_dump` базы данных.
2. Архивирует volume `appdata` (аудио-оригиналы, обложки, evidence-архивы для
   жалоб, база фингерпринтов Panako).
3. Кладёт оба файла в репозиторий `restic` (шифрованный, дедуплицированный).
   В проде — на Hetzner Storage Box по SFTP; локально/для тестов можно указать
   обычный путь на диске.
4. Удаляет старые снимки по политике: 14 ежедневных / 8 еженедельных / 6 ежемесячных.

Настройка (в `.env`):
```
RESTIC_REPOSITORY=sftp:u123456@u123456.your-storagebox.de:/backups/trackguard
RESTIC_PASSWORD=<сгенерировать: openssl rand -base64 32, сохранить в надёжном месте>
```
Пароль `RESTIC_PASSWORD` — **единственный ключ к бэкапам**. Если его потерять,
все резервные копии станут нечитаемыми навсегда. Храните его отдельно от сервера
(например, в менеджере паролей), не только в `.env` на той же машине.

Cron на VPS (пример, `crontab -e`):
```
0 3 * * * cd /opt/trackguard && bash scripts/backup.sh >> /var/log/trackguard-backup.log 2>&1
```

## Восстановление

```bash
bash scripts/backup.sh --list        # посмотреть доступные снимки
bash scripts/restore.sh latest       # восстановить последний снимок
```
`restore.sh` требует явного подтверждения (ввести слово `ВОССТАНОВИТЬ`) — команда
разрушительная, перезаписывает текущую базу и файлы.

## Репетиция восстановления — выполнена (приёмочный критерий M7)

**Дата:** 2026-07-20, устройство C:\SongGuard (второе dev-устройство проекта).

Поскольку это dev-машина без доступа к прод-Hetzner Storage Box, репетиция
проведена на **локальном restic-репозитории** (тот же restic, тот же формат —
единственное отличие от прода — расположение репозитория: локальный диск вместо
SFTP на Storage Box). Логика `backup.sh`/`restore.sh` идентична.

Шаги и результат:
1. `pg_dump` реальной dev-БД → 71 620 байт, 2104 строки SQL.
2. Архив `appdata` volume (`tar czf`) → 1.46 MiB (оригиналы аудио для Panako,
   база фингерпринтов Panako, обложки).
3. `restic backup` → снимок создан (`b5323827`), виден в `restic snapshots`.
4. `restic restore latest` в чистую директорию → оба файла восстановлены
   побайтово (проверено размером/содержимым).
5. SQL-дамп накачен в **отдельный одноразовый** контейнер PostgreSQL (не трогая
   рабочую dev-БД) → все 21 таблица на месте.
6. Проверены реальные данные после восстановления:
   - Треки (`HEAVENLY JUMPSTYLE`, ISRC `QZHN52501234`) — на месте.
   - Находки (`findings`) со статусами `removed`/`packet_ready` — на месте, с
     верным score/band.
   - Пакеты жалоб (`takedown_packets`) со статусами `sent`/`draft` и исходом
     `removed` — на месте.
   - Юридические данные пользователя (`legal_name`, `legal_email`) — на месте.
   - Архив `appdata` содержит `originals/*.wav` (эталоны Panako),
     `panako_dbs/` (индекс фингерпринтов), `covers/`.
7. Все временные контейнеры/директории репетиции удалены после проверки —
   рабочие dev-данные не были задеты.

**Вывод: механизм бэкап → восстановление работает end-to-end.** Единственное,
что не проверено с этого устройства — реальное SFTP-подключение к Hetzner Storage
Box (нужен купленный Storage Box и его SFTP-креды). Когда сервер и Storage Box
будут развёрнуты в проде, эту же репетицию стоит повторить один раз против
реального `RESTIC_REPOSITORY`, чтобы убедиться, что SFTP-доступ настроен верно.

## Что бэкапится, а что нет

**Входит в бэкап:** база данных целиком (все таблицы), `appdata` volume (аудио,
обложки, evidence-архивы, база Panako).

**Не входит:** код приложения (он в git, `git pull` при восстановлении сервера
с нуля), `.env` (секреты — храните их отдельную резервную копию сами, не в
общем restic-репозитории вместе с остальными данными, чтобы не смешивать
уровни секретности).
