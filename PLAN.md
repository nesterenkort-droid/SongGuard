# TrackGuard — система обнаружения пиратства музыкальных треков

> План для сборки субагентом. Язык UI: русский; код/комментарии: английский.
> Репозиторий: D:\NG (пустой, greenfield). Прошёл допрос пользователя (grill-me), веб-верификацию фактов и независимую архитектурную критику Plan-агентом.

## 1. Контекст

Автор треков (лейбл **0to8**) и коллеги страдают от пиратов, которые крадут готовые треки и **перевыпускают их через дистрибьюторов** (DistroKid и др.) — краденый релиз попадает сразу на Spotify, Apple Music и YouTube («Release - Topic» авто-каналы). Типичная пиратка: то же название + «(Slowed)» / «(Sped up)» / nightcore, часто то же имя артиста, та же обложка (иногда слегка изменённая нейронкой), чужой ℗-лейбл (например «℗ 13207436 Records DK» — автолейбл DistroKid). Бывают и прямые заливы на обычные каналы YouTube.

Золотой тест-кейс (реальный):
- Оригинал: «HEAVENLY JUMPSTYLE · TWXNY · Sxilwix · Innxcence», ℗ 0to8, 2025-11-28, видео `pSkW_rR65-g`
- Пиратки: «HEAVENLY JUMPSTYLE (Slowed) · TWXNY», Provided to YouTube by DistroKid, ℗ 13207436 Records DK, 2026-07-13 — видео `NzL0wDrGtYM`, `5bwI-Q0k31A`, `-HF6cbZ3KGg`

Нужна система: мониторит площадки, определяет пиратку «как человек» (детерминированные сигналы + аудио-фингерпринт + ИИ-судья для спорных), шлёт уведомления (Telegram-бот + веб-дашборд), готовит полуавтоматические пакеты жалоб.

## 3. Проверенные факты платформ (июль 2026)

1. **YouTube Data API**: у `search.list` отдельный бюджет ≈100 вызовов/день (правило июня 2026); расширение квоты — недели-месяцы. `videos.list`/`playlistItems.list` — 1 юнит. → Поиск экономим, каналы мониторим плейлистами почти бесплатно.
2. **Spotify API**: preview_url для новых приложений удалён (2024, отката нет). Поиск, лейбл альбома, **ISRC**, длительность — доступны (Client Credentials). Есть фильтр поиска `label:"..."`.
3. **iTunes Search API**: бесплатно, без ключа, 30-сек previewUrl, обложки, длительность; **НЕ отдаёт ISRC и лейбл** (лейбл Apple — только скрейпом веб-страницы; Apple Music API = $99/год — вне бюджета). Неофициальный rate-limit ~20 зап/мин/IP → медленная капельная подача.
4. **yt-dlp с датацентровых IP** часто ловит «Sign in to confirm you're not a bot» → скачивание аудио = необязательный этап обогащения; митигации: автообновление, cookies, паузы, прокси-резерв.
5. **Panako** (Java, AGPL): фингерпринт, устойчивый к time-stretch/pitch ±10%, **возвращает коэффициент растяжения**. Эталоны в вариантах 0.8/0.9/1.0/1.1/1.25 (ffmpeg asetrate/atempo) → покрытие −25%…+30%. ⚠️ Истинный stretch = вариант × репорт Panako (не забыть пересчёт!). AGPL: запускаем как неизменённый изолированный сервис, ссылка на исходники в футере.

## 4. Стек: Go vs Python — вердикт

**Python-монолит, и с отрывом.** (a) yt-dlp — это Python-*библиотека*: импорт даёт структурные ошибки и управление cookies/прокси, чего шелл из Go лишён; (b) aiogram, imagehash/Pillow, LLM SDK, Jinja/HTMX — родные; (c) нагрузка — сотни I/O-вызовов в **день**, не в секунду: преимущества Go не монетизируются, а JVM-сайдкар (Panako) существует при любом языке; (d) ИИ-субагенты собирают такой стек надёжнее всего. Где Go был бы лучше: публичный высоконагруженный сервис — не наш случай.

**Стек**: FastAPI + SQLAlchemy 2 + Alembic · PostgreSQL 16 · Redis 7 · **arq** (worker + cron; НЕ Celery — лишняя операционка) · Jinja2 + Tailwind (DaisyUI) + HTMX, тёмная тема · aiogram 3 · httpx · ffmpeg · yt-dlp (как библиотека) · Panako (JRE в образе worker) · **nginx** + certbot · Docker Compose · uv с lock-файлом, пиненные теги образов. Весь код асинхронный end-to-end (не смешивать sync/async). ИИ-судья: claude-haiku-4-5 (сборщику: перед интеграцией загрузить скилл `claude-api`).

## 5. Архитектура (Docker Compose)

```
nginx (:80/:443, TLS certbot, static) ──► web (FastAPI: UI + REST + auth)
                                           ├─ postgres · redis
worker (arq: pull-планировщик, сканеры, сигналы, скоринг,
```

Секреты в `.env`; конфиг детекции (веса, пороги, списки лейблов, расписания) — **в БД с админ-UI** (владелец не программист), изменения аудируются. Деплой: `deploy.sh` (git pull → build → up -d → alembic upgrade), плюс `update.sh`/`rollback.sh`.

### Авторизация без паролей
Вход через Telegram deep-link: «Войти через Telegram» → `t.me/<bot>?start=<nonce>` → бот подтверждает → сессия. Регистрация только при валидном инвайте. Бонус: каждый пользователь гарантированно подключён к боту. Админ-бутстрап: TG ID в `.env` + скрипт `promote.py` на сервере (страховка от потери админ-доступа).

## 6. Модель данных (ядро)

**Ключевой принцип: кандидаты — глобальные, находки — по трекам.** У коллег пересекающиеся каталоги (коллабы): один и тот же пиратский релиз не должен скачиваться дважды, пинговаться дважды и получать две DMCA.

- `users` (tg_user_id, display_name, is_admin, **legal_name, legal_address, legal_email** — для DMCA; без них пакет жалобы не генерируется) · `invites`
- `artists` (name, spotify_artist_id, apple_artist_id, yt_topic_channel_id) · `artist_members` (role owner/editor/viewer) · `track_artists` (m2m — коллабы)
- `tracks` (title, **normalized_title** (NFKC + confusables-fold), release_date, isrc, upc, duration_ms, spotify_track_id/album_id, apple_track_id, original_audio_path, **audio_ref_status: full/preview/none**, cover файл + **pHash и dHash**, **official_variants** (ISRC/ID легальных Slowed/Sped-версий — чтобы не флажить свои же), is_hot_pinned, per-platform last_scanned_at)
- `reference_fps` (track_id, variant_speed, panako-статус) — фингерпринты пересоздаваемы из хранимых аудио (скрипт rebuild)
- `whitelist_entries` — иерархия: scope global/tenant/track; типы: channel_id, **label_string (fuzzy: «℗ 0to8 under exclusive license…»)**, artist_id, ISRC, platform_id. Из находки — «внести в белый список в 1 клик»
- `pirate_entities` (тип: yt_channel / spotify_label / apple_label / distributor; first_seen, связи с находками) — watchlist + доказательства рецидива
- `platform_candidates` — **глобальные**, UNIQUE(platform, native_id): url, title, uploader, description_raw, parsed_provider, parsed_plabel, published_at, duration_ms, thumb, raw_json, first/last_seen
- `findings` — UNIQUE(candidate_id, track_id): score, band, **signals JSONB (сырое значение + нормированный вклад + вес + версия порогов — иначе не отладить и не натренировать)**, audio_match JSONB (matched, true_stretch_pct, offset, score), llm JSONB (verdict, confidence, reasoning_ru, cost)
- Жизненный цикл (enum + аудит переходов): `detected → scored → pending_review | remix_review → confirmed | dismissed | tolerated → packet_ready → sent → removed | still_alive | counter_noticed | reappeared(link)`. **tolerated** (фан-канал, которому разрешаем) ≠ dismissed: подавляет повторные флаги, но кандидат виден
- `finding_events` — полный аудит (кто/что/до/после; «кто нажал отправить DMCA» имеет юридический вес)
- `takedown_packets` (route: yt_form / spotify_form / apple_email / **distributor_email**, текст, evidence JSONB, sent by/at, outcome, **follow_up_at** — напоминание, если площадка молчит 14 дней)
- `evidence_archive` — снапшот **в момент подтверждения**: JSON API, обложка/миниатюра, отчёт Panako (после удаления пиратки доказать что-либо уже нельзя)
- `subscriptions` (user, artist, mode instant/daily/weekly, quiet hours) · `notification_outbox` (dedupe-ключи, retry — рестарт бота не дублирует и не теряет)
- `scan_jobs` + `quota_ledger` (дата, API, юниты, исход) — питают бюджетер, health-страницу и ответ на «почему трек не просканирован»
- `app_settings` (веса, пороги, списки своих лейблов, лимит LLM-расходов)

## 7. Конвейер детекции

### Ярус 0 — дифф страниц артиста (самый дешёвый и самый сильный; находка критики)
Пират, сохранивший имя артиста, почти всегда попадает **на ту же страницу артиста** в Spotify/Apple (DSP матчит по имени). Ежедневно: `GET /artists/{id}/albums` (Spotify, 1 вызов) и iTunes `lookup?id=&entity=album` → дифф с известным каталогом + official_variants → **любой неизвестный релиз = кандидат с высочайшим приором, ловится даже при полной смене названия**. Аналогично на YouTube: загрузки своего Topic-канала через playlistItems (1 юнит). Watchlist пиратских лейблов: поиск Spotify `label:"13207436 Records DK"` — весь новый выхлоп пирата одним запросом.

### Ярус 1 — поиск
YouTube: **один** комбинированный запрос «title artist» на трек (поиск нечёткий, «(Slowed)»-варианты всплывают сами; суффикс-варианты запросов = сожжённая квота). Spotify search track:"…", iTunes term=… с ротацией маркетов (US/DE/GB — региональные релизы). Дедуп по (platform, native_id).

### Ярус 2 — сигналы (все объяснимые, каждый пишется в signals JSONB)
- нормализация названий (NFKC, confusables/гомоглифы, zero-width, «s l o w e d») → exact/fuzzy совпадение; пиратский суффикс (slowed/sped/nightcore/8d/reverb/…)
- совпадение артиста; **соотношение длительностей** (candidate/original ≈ 1/0.8, 1/1.25 → почти вердикт без аудио; метаданные бесплатны на всех трёх площадках)
- **ISRC**: совпал с нашим → авто-легально (наша же доставка); тот же title+artist с ДРУГИМ ISRC → сильный пиратский сигнал
- парсинг описания YT: «Provided to YouTube by X», «℗ Y». Пользователь подтвердил: **0to8 никогда не лился через DistroKid** → «DistroKid» на нашем треке = сильный пиратский сигнал сам по себе. Общий механизм: каждый tenant декларирует список СВОИХ дистрибьюторов/лейблов (fuzzy-сверка, у 0to8 — только «0to8»); provider/℗ вне списка = чужое. Регэксп `^\d+ Records DK$` и совпадение с pirate_entities — максимальный приор
- дата: подписанная дельта как **взвешенный сигнал, не фильтр** (утечки выходят и ДО официального релиза)
- pHash+dHash обложки (для YT-миниатюр — center-crop квадрата из letterbox)
- гейт: канал/ISRC/лейбл из whitelist → это наш релиз, не кандидат

### Ярус 3 — скоринг → аудио → ИИ
Стартовые веса (правятся в админке, версия порогов пишется в находку): title exact +40 / fuzzy +30; суффикс +15; артист +20; duration-ratio на пресете ±2% +30; чужой ISRC при матче метаданных +15; provider/лейбл вне tenant-списка +20 / DistroKid на треке 0to8 +25 / из watchlist или `\d+ Records DK` +25; pHash≤8 +25 (≤14 +15); дельта дат ±5; аудио-матч +40 / НЕ-матч −50. Пороги: ≥70 high → находка сразу; 40–69 mid → аудио-проверка; <40 → лог.
Аудио: кандидат — iTunes preview (DSP) или yt-dlp `--download-sections` 60 сек (YT, best-effort с cookie/proxy-фолбэком) → ffmpeg wav 16k mono → Panako → matched + **true_stretch = вариант × репорт**. Эталон трека: приоритет upload WAV > yt-dlp собственного Topic-видео > iTunes preview собственного трека (30-сек эталон = слабый, риск невнахлёста с 30-сек запросом; статус эталона виден в UI, система «пилит» пользователя догрузить оригинал). Нет эталона → работаем на метаданных+обложке+ИИ.
ИИ-судья (mid-band остаток + все «ремиксы»): claude-haiku-4-5, JSON {verdict, confidence, reasoning_ru}; вход — таблица сигналов обеих сторон; **жёсткий месячный лимит расходов в коде**: при исчерпании — очередь на ручную проверку, никогда молчаливый дроп.

### Кросс-триггеры и жизненный цикл
Находка на DSP → точечная проверка того же названия на YT (и наоборот), **окно дедупликации триггеров 24 ч на (track, platform)** — иначе петля. Подтверждённые находки перепроверяются раз в N дней → removed («победа!» в ТГ) / still_alive / reappeared (новая залив тем же каналом линкуется в цепочку — доказательство рецидива для страйков).

## 8. Планировщик и бюджетер (pull-модель; переработано по критике)

- **Pull, не cron-push**: каждые 15 мин планировщик вычисляет «долг»: `tracks where last_scanned_at < cutoff яруса order by priority limit остаток_бюджета`. Самовосстанавливается после ребутов/простоев, нет гонок расписаний. Один лидер (Redis-lock).
- **Бюджет YouTube-поиска**: атомарный леджер в Redis (Lua check-and-spend), дефолт 90/день, **сутки по полуночи Pacific** (так ресетится Google, не серверный TZ!). Резервные полы: manual ~10 > кросс-триггеры ~20 > hot > ротация (вытесняемая, возобновляемый курсор). Математика сходится только при **одном запросе на трек** (см. §7 Ярус 1). Watchlist и Topic-каналы — вне поискового бюджета (playlistItems). Purge-контроль: p95 возраста скана > 10 дней → алерт админу.
- **Hot-список с распадом**: свежие релизы <60 дней, ранее пиратившиеся, закреплённые. Демоция после N чистых сканов / 30 дней без находок (иначе hot разрастается до всего каталога и душит ротацию); ручные пины не распадаются.
- **Блокировки in-flight** SETNX+TTL на (track, platform) — триггер и ротация не сканируют одно дважды; UNIQUE-констрейнты findings как последний рубеж.
- **Пер-платформенные очереди** с независимой конкуренцией и token-bucket (капля Apple 20/мин не блокирует Spotify), circuit-breaker на 429/quotaExceeded — платформа паркуется, не долбится.
- **Первичный импорт нового пользователя** (300 треков разом) идёт через тот же бюджетер: DSP — за часы, YouTube — за дни; UI показывает «первичное сканирование 40%, ETA вторник» (иначе владелец сочтёт это багом).

## 9. Telegram-бот

Карточка находки: обложка пиратки, трек/артист, площадки, канал/лейбл пирата, дата, скор + сигналы человеческим языком («замедлено на 18%», «обложка совпадает на 97%», «лейбл: 13207436 Records DK», «длительность 1:0.8 — типичный slowed»), кнопки [✓ Пиратка] [✗ Ложное] [🕊 Разрешить каналу] [📄 Пакет жалобы] [🌐 На сайте]. /check (inline-выбор трека) · /status (каталог, находки, квота, здоровье) · дайджесты instant/daily/weekly + quiet hours · через outbox (без дублей при рестарте). **Админ-алерты** (только смена состояния, простым языком «что это значит / что делать», + опц. ежедневный «всё ок» для доверия — без alert fatigue): упал скан, квота, бот-стены yt-dlp, протух ключ, диск 80%, бэкап старше 26 ч.

## 10. Веб-интерфейс (SSR, «красиво», RU)

Дашборд-лента (фильтры статус/артист/платформа/скор, разбор сигналов у каждой находки) · Каталог (импорт по ссылке, drag&drop оригиналов, статус аудио-эталона, pin) · Трек (история) · Находка (сигналы, обложки рядом, аудио-доказательство, таймлайн, пакет) · Артисты/участники · Белые списки / Watchlist пиратов · Настройки (ключи, веса, лимит LLM) · Админка (инвайты, пользователи, здоровье с плитками «последний успешный вызов» по каждому ключу, квоты-леджер, качество детекции по диапазонам скора).

## 11. Takedown-пакеты (полуавтомат)

Шаблоны EN + RU-пояснения. Требуются legal-поля пользователя (имя, адрес, email) — без них генерация блокируется. Для коллабов **ровно один владелец жалобы** (дубль-DMCA от «двух правообладателей» = приглашение к контр-нотису). Маршруты: **(1) дистрибьютор — приоритетный** (жалоба DistroKid abuse убивает релиз на ВСЕХ площадках и может грохнуть аккаунт пирата), (2) YouTube copyright webform, (3) Spotify форма, (4) Apple (контент-диспут, email). Evidence-архив снимается в момент подтверждения. follow_up_at: нет ответа 14 дней → напоминание. ⚠️ Актуальные URL форм проверить на этапе M6. Автоотправки нет.

## 12. Эксплуатация (владелец — не программист)
